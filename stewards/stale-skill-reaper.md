# StaleSkillReaper — first steward

## What is a steward?

A steward is a manually- or cron-triggered agent that reads the global work graph (`~/.claude/work-graph.jsonl`) and writes recommendations to the global inbox (`~/.claude/inbox.md`). Stewards are **outside the Stable Core** — the crowd may add, remove, or modify stewards freely. They are the v2.0 mechanism for self-improvement *across* runs (Phase 9 Roster Review handles *within-run* self-improvement).

Stewards are invoked as `/forge steward <name>`. The orchestrator looks up `stewards/<name>.md` and dispatches a single agent with this file as its prompt. No worktree is created; no brief is required.

## Inputs

- `~/.claude/work-graph.jsonl` — global event log written by Phase 11 (Observe). Missing file → treat as empty graph.
- `~/.claude/skills/.history.jsonl` — Phase 9 Roster Review audit journal. Missing file → treat as empty history.
- `~/.claude/skills/` — current skill roster on disk. Missing directory → exit `DONE` with `## No skill roster found`.
- Optional override: `STALE_DAYS=<int>` environment variable. Default: `60`.

## Output

`~/.claude/inbox.md` — fully rewritten on each run. Format:

```markdown
# Forge Inbox

> Generated by stewards. Triage by editing this file. The forge does not auto-apply recommendations in v2.0.

## Stale skill: <skill-name>
- last_used: <ISO8601 timestamp> (<N> days ago)
- run_count_90d: <int>
- recommendation: fire (no usage in <STALE_DAYS> days)
- evidence:
  - <event-1 summary>
  - <event-2 summary>
  - <event-3 summary>

## Stale skill: <skill-name>
...
```

If no skills are stale, emit a single section:

```markdown
# Forge Inbox

## No stale skills

All skills in the current roster have been used within the last <STALE_DAYS> days. Nothing to triage.
```

## Schedule

Manual trigger only in v2.0. v2.1+ will add cron-driven runs via a sidecar daemon. To run manually:

```
/forge steward stale-skill-reaper
```

## Recommended model

haiku — this is grep + jq + arithmetic, not reasoning. If the model tier ever escalates above haiku for this steward, something has gone wrong (probably the rule logic creeping into LLM territory — fix the rule, not the model).

## Hard rules

- **Never ask the user a question.** Ambiguous skill names → emit them all; the user triages.
- **Never delete skills.** v2.0 is recommend-only. The recommendation is `fire` (the noun, not the verb). Removal is the user's call.
- **Read-only on `~/.claude/skills/`.** This steward only writes to `~/.claude/inbox.md`.
- **Idempotent.** Two consecutive runs without intervening `/forge` invocations produce byte-identical output.
- **Time-box at 30 seconds total wall clock.**
- **Status:** `DONE` (inbox written) | `DONE_NO_DATA` (empty graph + empty history → exit clean, inbox unchanged) | `BLOCKED` (cannot read inputs or write inbox).

## Step 1 — Collect skill universe

```bash
# All skills currently on disk
ls -1 ~/.claude/skills/ 2>/dev/null | grep -v '^\.' > /tmp/forge-skills-roster.txt

# All skills ever mentioned in the work graph
jq -r 'select(.type == "phase_completed") | .payload.name' ~/.claude/work-graph.jsonl 2>/dev/null | sort -u > /tmp/forge-skills-from-graph.txt

# All skills ever mentioned in the Phase 9 audit journal
jq -r '.skill // empty' ~/.claude/skills/.history.jsonl 2>/dev/null | sort -u > /tmp/forge-skills-from-history.txt
```

If `/tmp/forge-skills-roster.txt` is empty: emit `## No skill roster found` and exit `DONE_NO_DATA`.

## Step 2 — Compute last-used timestamp per skill

For each skill name in the roster, find the most recent timestamp across both sources:

```bash
for skill in $(cat /tmp/forge-skills-roster.txt); do
  last_graph=$(jq -r --arg s "$skill" \
    'select(.type == "phase_completed" and .payload.name == $s) | .ts' \
    ~/.claude/work-graph.jsonl 2>/dev/null | sort | tail -1)

  last_history=$(jq -r --arg s "$skill" \
    'select(.skill == $s) | .ts' \
    ~/.claude/skills/.history.jsonl 2>/dev/null | sort | tail -1)

  # Pick whichever is newer; empty if both empty
  last=$(printf '%s\n%s\n' "$last_graph" "$last_history" | sort | tail -1)

  echo "$skill|$last"
done > /tmp/forge-skills-last-used.txt
```

## Step 3 — Classify

Compute the cutoff timestamp:

```bash
STALE_DAYS="${STALE_DAYS:-60}"
cutoff=$(date -u -d "$STALE_DAYS days ago" +%Y-%m-%dT%H:%M:%SZ 2>/dev/null \
  || date -u -v-${STALE_DAYS}d +%Y-%m-%dT%H:%M:%SZ)
```

For each skill in `/tmp/forge-skills-last-used.txt`:

- Empty `last` → **stale** (never used).
- `last < cutoff` → **stale**.
- `last >= cutoff` → **fresh**.

## Step 4 — Build evidence for each stale skill

For each stale skill, collect the **last 3 events** (graph + history merged, newest last) as evidence. If the skill has 0 events, evidence is `never observed`.

## Step 5 — Skills in the Stable Core are exempt

Read `STABLE_CORE.md` from the agent-forge install root (`~/.claude/skills/workflow-agents-loop/STABLE_CORE.md` or the project-level equivalent). Skills listed there are NEVER recommended for firing, regardless of usage. Surface them in a separate section if they appear stale:

```markdown
## Stable Core skill never observed: <name>

The skill `<name>` is in the Stable Core but has no entries in the work graph or audit journal. This usually means: (a) the graph is too young (< 60 days of runs), or (b) the orchestrator is not dispatching this skill correctly. Investigate; do not fire.
```

## Step 6 — Write the inbox

Rewrite `~/.claude/inbox.md` from scratch with:

1. The `# Forge Inbox` header + the one-line preface.
2. One `## Stale skill: <name>` section per stale skill from Step 3, with the evidence from Step 4.
3. One `## Stable Core skill never observed: <name>` section per Stable Core skill from Step 5.
4. If no stale skills and no missing Stable Core skills: the `## No stale skills` section from the Output spec.

## Step 7 — Status

Output to caller:

```
DONE — StaleSkillReaper complete. Wrote <N> stale recommendations and <M> Stable Core notes to ~/.claude/inbox.md.
```

If `~/.claude/work-graph.jsonl` and `~/.claude/skills/.history.jsonl` are both empty AND the roster is empty:

```
DONE_NO_DATA — Nothing to analyze yet. Run /forge a few times to populate the work graph.
```

## Why this is the first steward

StaleSkillReaper is the cheapest steward to build that proves the v2 architecture works end-to-end:

- It exercises **graph reads** (proves Phase 11 writes are useful).
- It exercises **roster reads** (proves the steward can navigate `~/.claude/skills/`).
- It exercises **inbox writes** (proves the user-facing surface works).
- It's **fully deterministic** (rule-based, no LLM judgment) — easy to test, easy to verify, easy to trust.

If this steward works, the architecture works. The four remaining stewards (Sentinel, ConfigAuditor, PhaseROI, ChangeGate) are variations on the same pattern with different query logic.
